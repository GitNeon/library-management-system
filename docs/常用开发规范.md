这是一份基于 **Google C++ Style Guide** 提炼整理的业务开发核心规范。

在业务开发中，代码的可读性和可维护性往往是第一优先级。Google 风格的核心在于通过**统一的命名**（如变量名 `snake_case`，类名 `PascalCase`）和**严格的格式**（如 `const` 引用传参）来降低团队协作的沟通成本。以下内容涵盖了日常开发中最高频使用的规范，你可以直接作为团队手册使用。

---

# Google C++ 开发规范（业务开发核心版）

> 基于 Google C++ Style Guide
> 目的：统一代码风格，保证可读性与可维护性，减少 Code Review 中的无意义争论。

## 1. 头文件管理

**核心原则：让头文件自给自足，尽量轻量。**

### 1.1 Include 保护
所有头文件必须使用 `#define` 保护，格式为：`<PROJECT>_<PATH>_<FILE>_H_` 。
```cpp
#ifndef MYPROJECT_BASE_LOGGING_H_
#define MYPROJECT_BASE_LOGGING_H_
// 代码内容
#endif // MYPROJECT_BASE_LOGGING_H_
```

### 1.2 Include 顺序
为了隐藏依赖，**核心原则**是：`你的 .cc` 对应的 `.h` 放在第一位。
**推荐顺序**：
1. **本类对应的头文件** (`.h`)
2. **C 系统头文件** (如 `<stddef.h>`)
3. **C++ 标准库头文件** (如 `<vector>`)
4. **其它库的头文件** (如 `"third_party/..."`)
5. **本项目内的头文件** (如 `"base/logging.h"`)

```cpp
// foo.cc 示例
#include "foo.h"        // 必须第一个，用于检查 foo.h 是否自包含

#include <string>       // 标准库
#include <vector>

#include "base/logging.h" // 项目内头文件
```

## 2. 命名规范

**命名三原则：描述性强、避免缩写、风格统一。**

| 类别 | 风格 | 示例 |
| :--- | :--- | :--- |
| **文件名** | `snake_case` | `http_server.cc`, `url_utils.h` |
| **类 / 结构体 / 枚举** | `PascalCase` (大驼峰) | `class HttpServer`, `struct UrlInfo` |
| **函数 / 方法** | `PascalCase` (大驼峰) | `void ProcessRequest()` |
| **变量 (普通/成员)** | `snake_case`， 成员末尾加 `_` | `int table_count`; <br> `int table_count_;` |
| **常量 / 枚举值** | `k` + `PascalCase` | `const int kMaxRetry = 3;` |
| **宏定义 (尽量避免)** | `UPPER_SNAKE_CASE` | `#define LOG(severity) ...` |
| **命名空间** | `snake_case` | `namespace my_project ` |

### 2.1 实战案例
```cpp
// 好的命名
class NetworkService {                      // 类：大驼峰
public:
    void ConnectToServer();                 // 函数：大驼峰
private:
    int retry_count_;                       // 成员变量：下划线 + 尾下划线
    static const int kTimeoutSeconds = 30;  // 常量：k开头
};

int request_id = 0;                         // 普通变量：下划线
```

## 3. 代码格式

**核心工具：使用 `clang-format` 自动化格式化。**

### 3.1 缩进与行宽
*   **缩进**：**2个空格**（Google 标准）。**严禁使用 Tab 制表符**。
*   **行宽**：**80 或 100 字符**。超过则需换行。

### 3.2 大括号与空格
*   **函数**：大括号不另起一行。
*   **控制语句**：必须使用大括号，哪怕只有一行。

```cpp
// 正确示例
if (condition) {                // 圆括号与花括号之间有一个空格
    DoSomething();              // 缩进 2 空格
} else {                        // else 与前后大括号同行
    DoOtherThing();
}

// 错误示例
if (condition)
    DoSomething();              // 禁止：没有大括号

// 函数示例
void MyFunction() {             // 左大括号跟在函数名后面，空格隔开
    int x = 0;                  // 内部缩进 2 空格
}
```

### 3.3 指针与引用
`*` 和 `&` 紧跟类型，还是变量名？**两者皆可，但必须保持项目内一致**。推荐紧跟类型（强调类型）。
```cpp
// 推荐写法
int* ptr = &x;          // 星号靠左
const string& name = y;

// 注意：不能同时声明指针和变量
int* a, b;              // 错误：b 是 int 不是指针
int* a = nullptr;       // 正确
int* b = nullptr;       
```

## 4. 语言特性规范

**这些是业务开发中最容易踩坑的地方。**

### 4.1 引用参数
所有以**引用传递且不会被修改**的参数，必须加 `const`。
```cpp
// 正确：只读
void DisplayInfo(const UserInfo& user);

// 正确：输出参数（指针，且通常非空）
void FillDetails(UserInfo* user);

// 错误：非常量引用，调用者无法得知数据会被修改
void FillDetails(UserInfo& user);  // 禁止
```

### 4.2 类型转换
**禁止使用 C 风格的 `(int)x`** 。必须使用 C++ 的转换，便于编译器检查。
*   `static_cast`: 最常用，如数值转换、基类转子类（不安全但常用）。
*   `dynamic_cast`: 仅用于多态继承的安全向下转型（需要 RTTI）。
*   `const_cast`: 去掉 const 属性（**业务开发中极不推荐**）。
*   `reinterpret_cast`: 底层转换，如指针转 int（**非常危险，禁用**）。

```cpp
float a = 1.0f;
int b = static_cast<int>(a);        // 正确
int c = (int)a;                     // 禁止
```

### 4.3 智能指针
*   **禁止使用原始指针管理所有权**。
*   `std::unique_ptr`：**首选**。独占所有权，开销极小。
*   `std::shared_ptr`：谨慎使用。引用计数有性能开销，且容易产生循环引用导致内存泄漏。

### 4.4 注释
代码告诉你怎么做，注释告诉你为什么这么做。
*   **文件头**：简要描述该文件的作用。
*   **类/函数**：使用 `//` 描述功能和参数含义。
*   **TODO**：临时代码必须加 `// TODO(author): 说明`。

## 5. 业务开发黄金法则

1.  **const 狂热**：能加 `const` 的地方尽量加（成员函数、局部变量、参数）。
2.  **使用 `nullptr`**：**禁止**使用 `NULL` 或 `0` 表示空指针。
3.  **避免宏**：能用 `constexpr` 或 `const` 解决的问题，就不要用 `#define`。
4.  **auto 简化**：仅在类型显而易见时使用 `auto`（如迭代器、`auto ret = func();`），不要滥用。
5.  **尽量简单**：不要为了“炫技”使用复杂的模板元编程或 Boost 库，除非业务确实需要。
